## 1. Core Architecture
The app follows a multi-layered approach to tunnel traffic:
- **Local Interception**: The app uses Android's `VpnService` to capture all device traffic and route it through a TUN interface.
- **Local Proxy & Injector**: Traffic is then passed to a local proxy (usually port 7878) which uses an "Injector" to modify the packets.
- **Tunneling Engine**: The modified packets are sent through an encrypted tunnel (Psiphon, Xray, or OpenVPN) to a remote server.
- **Server-Side**: The remote server receives these "tricked" packets, strips the injection headers, and fulfills the original internet request.

## 3. How "Free Internet" is Achieved

### The "Bug" or "Bypass" Principle
Base station firewalls (at the ISP level) often have "White Lists" of websites that are accessible even with zero balance (e.g., the telecom's website, WhatsApp, or Facebook on certain plans).

MX-Tunnel exploits this by:
1. **SNI Spoofing**: When starting a TLS connection, the app puts a whitelisted domain (the "SNI") in the packet header. The firewall sees `m.facebook.com` and lets it through, but the actual encrypted data inside is destined for the VPN server.
2. **HTTP Header Injection**: For non-HTTPS traffic, the app injects a `Host: whitelisted-domain` header.
3. **Tunneling**: Once a connection is established between the app and the VPN server (using the whitelisted host as a "shield"), all other traffic (YouTube, Gaming, Web Browsing) is encapsulated inside this connection.

## 4. Server Side Role
While we only see the client code, the server side must be:
- **Proxy Aware**: It expects traffic arriving with these "injected" headers.
- **Payload Stripping**: It decapsulates the packets to see the *real* destination (e.g., `google.com`).
- **Reverse Proxying**: It fetches the data from the real destination and sends it back through the same "tricked" tunnel to the client.
